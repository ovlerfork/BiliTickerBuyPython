# Release Policy

Release branches are generated, not edited by hand.

- `main` follows upstream `main`; `.github/workflows` is removed before pushing to this fork.
- `latest` is generated from `main` or `PATCH_BASE_REF` plus `patches/cur/*.patch`; `.github/workflows` is removed before pushing.
- Release tags use `v<pyproject-version>-patch<N>`.
- `patch1` is the first release for a new `pyproject.toml` version and is generated by CI when upstream bumps that version.
- `patch2` and later for the same version are prereleases.

Every patch release publishes:

- Exact Docker tag: `ghcr.io/<owner>/<repo>:v<pyproject-version>-patch<N>`.
- Rolling Docker tag: `ghcr.io/<owner>/<repo>:patch`.

`scripts/detect-patch-version.sh` reads `[project].version` from `pyproject.toml`, checks local and remote tags, and emits shell assignments plus GitHub output values when `GITHUB_OUTPUT` is set.

`scripts/publish-latest.sh` is the CI entry point for generating `latest`, computing the next patch tag, writing release notes data, pushing the generated branch and tag, and creating the GitHub Release when `GH_TOKEN` or `GITHUB_TOKEN` is available. It refuses to push unless `CI` is set or `ALLOW_BRANCH_PUSH=1` is provided.
